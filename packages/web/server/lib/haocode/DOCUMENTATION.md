# HaoCode compatibility runtime

This module keeps the OpenChamber browser contract stable while replacing the upstream OpenCode process with HaoCode.

- `store.js` persists adapter-owned sessions, OpenCode-shaped message snapshots, pending interrupts, provider settings, and status.
- `worker-supervisor.js` owns one short-lived PHP process per active session. The PHP process is not an HTTP sidecar; it reads one request over stdin and emits JSON-lines events over stdout. Inside the worker, fresh runs execute through the SDK's first-class `Agent`/`Runner` API (`Runner::stream`); session continuations (requests with `haocodeSessionId`) and interrupt resumes stay on the `HaoCode` facade because `RunOptions` carries no session id and `Agent` cannot carry the extra provider request headers some providers require.
- `compat-server.js` exposes the subset of the OpenCode HTTP/SSE protocol required by the existing UI and translates HaoCode worker events into OpenCode session events.

Project and user Agent/Command definitions remain stored in OpenCode-compatible config files so the inherited settings UI can edit them. The adapter lists those definitions through `/agent` and `/command`; the selected agent is forwarded to the PHP worker as a structured `agent: { name, prompt?, model?: { providerID, modelID } }` field on the run request (only configured keys are included, so the default `build` agent sends just `{ name: 'build' }`). The worker binds it to an SDK `Agent`: `name` becomes the agent name, `prompt` keeps flowing through the SDK's `appendSystemPrompt` slot — appended to HaoCode's default coding system prompt and never mapped to `systemPrompt`, which would replace that default wholesale — and a pinned `model.modelID` (accepted as an object or an OpenCode-style `provider/model` string) overrides the run's model id while provider credentials still come from the request's provider section. MCP definitions are converted to a private, generated HaoCode settings file for each project and enabled for that worker run. `/mcp` reports whether each definition is enabled and structurally usable; the actual connection is established by HaoCode when a run starts.

HaoCode `TodoWrite` events are persisted per session and published as `todo.updated`. Session diff, VCS diff/status, and file status responses are derived from the selected workspace's current Git worktree. They describe current workspace changes, including changes that predate the session; they are not an attribution ledger. HaoCode's LSP client is created only while the `LSP` tool runs, so `/lsp` correctly has no persistent server process to report even though the tool is available to the agent.

Message history pagination preserves complete renderable turns: a limited page includes any older parent user messages referenced by assistant records in that page, and `x-next-cursor` identifies the next older base page. Supplemental parent records may therefore make a response slightly larger than its requested limit.

Session revert is a session-level marker, not a file restore: `POST /session/:sessionID/revert` requires `messageID` (400 when missing, 400 `Message not found` when the id is not in the session's stored messages, 404 for unknown sessions) and records `session.revert = { messageID, partID? }`; `POST /session/:sessionID/unrevert` deletes it. Both bump `time.updated`, publish `session.updated`, and return the session. The UI derives hidden messages from this marker; this server keeps no snapshots, so workspace files are never rolled back.

Revert visibility is lexicographic: the renderer keeps messages with `id < revert.messageID` and generates user-message IDs as `msg_<12-hex (timestamp*0x1000+counter)>`. Server-persisted message IDs must interleave with that scheme by time, so `store.js` exposes `createMessageId()` (same 6-byte big-endian layout, counter starting at `0x800` so a server message created in the same millisecond as its user prompt sorts after it); never use the base36 `createId('msg')` for stored messages — base36 compares greater than the hex scheme and a revert would hide every server-generated message.

The worker supervisor spawns PHP with `-d display_errors=stderr` because the packaged php-bin runtime has no php.ini and defaults `display_errors` to STDOUT, which would let any PHP warning/notice corrupt the JSON-lines protocol channel; stderr is captured for diagnostics instead. As defense in depth, stdout lines matching `^(PHP )?(Warning|Notice|Deprecated|Strict standards)` are skipped rather than parsed, counted, and appended to a non-zero exit's error message as `(N PHP warning/notice line(s) skipped)`.

Image attachments follow a per-provider input policy persisted as `state.providers[id].imagePolicy` (`'native' | 'ocr' | 'caption' | 'vlm' | 'drop'`, default `'native'`) plus `state.providers[id].imageVlmModel` (string or `null`, required when the policy is `vlm`), both managed through `PATCH /provider/:id/settings` (illegal policy values and `vlm` without a model are rejected with 400) and reported by the settings GET. Individual models can override the provider default through `state.providers[id].modelImagePolicies`, a `{ [modelID]: policy }` map also managed through the same PATCH: send `modelImagePolicy: { model: string, policy }` to set one model's override and `modelImagePolicy: { model: string, policy: null }` to clear it (an empty map deletes the field); the settings GET reports the sanitized map. The effective policy for a prompt is `modelImagePolicies[modelID] ?? imagePolicy ?? 'native'`; a `vlm` override requires the provider-level `imageVlmModel` (still the single VLM model id) and is rejected with 400 without it. When a prompt carries image parts and the policy is non-native, prompt_async converts each image to text before sending and withholds the `images` field from the worker request: `ocr` runs RapidOCR (PP-OCRv4 Chinese+English models bundled via `@gutenye/ocr-models`, executed on onnxruntime-node — fully offline, no downloads), `caption` is a combined pass: RapidOCR extracts embedded text and `Xenova/vit-gpt2-image-captioning` (through `@xenova/transformers`, weights lazily cached to `<dataDir>/haocode/models/`; the import hides Bun's `self` global so the library selects its Node/sharp image-decoding branch) describes the scene — each half that succeeds contributes a `[识别文字]` / `[画面描述]` section, and only a double failure degrades, and `vlm` asks the configured vision model on the provider's own endpoint (OpenAI chat/completions with an `image_url` part, or Anthropic `/v1/messages` with a base64 image block when `providerType` is `anthropic`). Each converted image is appended to the prompt as `[图片 {filename 或序号}: {转换文本}]`; a failed or 60-second-timed-out conversion degrades to `[图片 …: 图片转述失败]` without blocking the run, conversions run concurrently via `Promise.all`, and the RapidOCR instance / caption pipeline are singletons reused across runs. `drop` simply discards the attachments (the prompt keeps the `[Attached image name]` note), and `native` preserves the previous behavior of forwarding data URIs untouched.

Model limits have one runtime source of truth. The compatibility server resolves each model's context and output limits from the shared models.dev catalog, exposes those values through `/config/providers`, and sends the same values to the PHP worker. When the catalog is unavailable and no stale cache exists, it falls back to HaoCode's `200000` context and `16384` output defaults. The worker applies the resolved context through `HAOCODE_CONTEXT_WINDOW`, so the provider page, Context sidebar, and HaoCode compaction budget use the same limits.

Built-in Anthropic and OpenAI support OAuth subscription login alongside API keys; `GET /provider/auth` advertises one `{ type: 'oauth' }` method for each (all other built-in presets and custom providers stay api-key only). Both flows use PKCE (S256) implemented with `node:crypto` — no extra dependencies. Anthropic uses the manual paste-code flow (`method: 'code'`): `POST /provider/anthropic/oauth/authorize` returns the `claude.ai/oauth/authorize` URL whose `state` doubles as the PKCE verifier, and `POST /provider/anthropic/oauth/callback` accepts the pasted `"<code>#<state>"` (or a bare code, which reuses the pending state), exchanges it at `console.anthropic.com/v1/oauth/token`, and stores the result. OpenAI uses the browser flow (`method: 'auto'`): authorize starts a `node:http` localhost redirect listener (ports 1455-1465 tried in order), and the callback route awaits the browser redirect (120s timeout), validates the returned state, exchanges the code at `auth.openai.com/oauth/token`, and extracts `accountId` from the `chatgpt_account_id` JWT claim when present. Tokens persist as `state.providers[id].oauth = { access, refresh, expires (ms epoch), accountId? }` in the mode-`0600` runtime-state file and are never returned by API responses; pending flows live only in memory and expire after 10 minutes. Credential resolution for worker runs is oauth access token → saved apiKey → environment variable: a token within 5 minutes of expiry is refreshed synchronously through the provider's `refresh_token` grant first (single-flight per provider, written back to state; a failed refresh falls back to the saved apiKey or environment). OAuth client ids default to the public OpenCode clients and can be overridden through `HAOWORK_OAUTH_ANTHROPIC_CLIENT_ID` / `HAOWORK_OAUTH_OPENAI_CLIENT_ID`. When the resolved credential is an Anthropic OAuth access token the worker request marks it `oauthBearer: true`, which `worker.php` forwards as `HAOCODE_OAUTH_BEARER=1` because `HaoCodeConfig` has no such constructor field. `DELETE /auth/:id` removes both the saved apiKey and the oauth record and cancels any pending flow for that provider.

GitHub Copilot is a built-in preset defined outside `PROVIDER_DEFINITIONS` (separate change partition) and merged into the registry as a built-in; its OAuth login is a dual-stage device-code flow (`method: 'auto'`, label `GitHub 账号（设备码授权）`). Stage one: `POST /provider/github-copilot/oauth/authorize` requests a device code from `github.com/login/device/code` (public VS Code client id `Iv1.b507a08c87ecfe98`, scope `read:user`, overridable via `HAOWORK_OAUTH_GITHUB_COPILOT_CLIENT_ID`) and returns `{ url: verification_uri, method: 'auto', instructions: 'Enter code: <user_code>' }`, which the generic auto-flow UI renders; `POST /provider/github-copilot/oauth/callback` (no code) polls `github.com/login/oauth/access_token` with the `urn:ietf:params:oauth:grant-type:device_code` grant — `authorization_pending` waits `max(1s, interval)` between polls, `slow_down` grows the interval by 2s, `expired_token`/`access_denied` fail the flow, and the total budget is 15 minutes. Stage two: the granted GitHub token is exchanged for a short-lived Copilot API token at `api.github.com/copilot_internal/v2/token` (GET with `Authorization: Bearer`, `Editor-Version: vscode/1.96.2`, `X-Github-Api-Version: 2025-04-01`; `expires_at` is a seconds epoch, millisecond epochs tolerated). The record persists as `state.providers['github-copilot'].oauth = { access: <copilot token>, refresh: <github token>, expires }`; there is no refresh_token grant, so a token inside the 5-minute expiry skew is "refreshed" by re-running the stage-two exchange with the stored GitHub token (single-flight, guarded write-back). The saved apiKey and the env aliases (`COPILOT_GITHUB_TOKEN`, `GH_TOKEN`, `GITHUB_TOKEN`, first set wins) are also GitHub tokens, not Copilot credentials — the Copilot API rejects GitHub tokens directly — so they go through the same exchange on first use and the result is cached under `state.oauth` for later runs. Every github-copilot worker request carries `provider.headers` with the Copilot API headers (`Editor-Version: vscode/1.96.2`, `Editor-Plugin-Version: copilot-chat/0.35.0`, `User-Agent: GitHubCopilotChat/0.26.7`, `Copilot-Integration-Id: vscode-chat`, `Openai-Organization: github-copilot`), which `worker.php` forwards verbatim to `HaoCodeConfig`'s `headers` named parameter. The preset ships a static model list (`gpt-4o`, `gpt-4.1`, `o3-mini`, `claude-sonnet-4.5`) rather than querying the Copilot catalog; note that Claude models on Copilot speak the Anthropic protocol while this preset sends everything as `openai_chat`, so the Claude entries may not work in v1.

The provider registry merges built-in definitions with user-defined custom providers persisted in `state.customProviders[id] = { name, providerType, baseUrl, env, models, contextWindow?, maxTokens?, createdAt }`. The built-in presets are Anthropic, OpenAI, DeepSeek, and twenty more services registered in `PROVIDER_DEFINITIONS` — OpenRouter (`OPENROUTER_API_KEY`), xAI (`XAI_API_KEY`), Groq (`GROQ_API_KEY`), Mistral (`MISTRAL_API_KEY`), Moonshot AI/Kimi (`MOONSHOT_API_KEY`, `KIMI_API_KEY`), Z.AI/GLM (`ZAI_API_KEY`, `Z_AI_API_KEY`), Qwen/DashScope (`DASHSCOPE_API_KEY`), Together AI (`TOGETHER_API_KEY`), Fireworks (`FIREWORKS_API_KEY`), Cerebras (`CEREBRAS_API_KEY`), Hugging Face (`HF_TOKEN`), Volcano Engine Ark (`VOLCANO_ENGINE_API_KEY`), MiniMax (`MINIMAX_API_KEY`), Baidu Qianfan (`QIANFAN_API_KEY`), SiliconFlow (`SILICONFLOW_API_KEY`), StepFun (`STEPFUN_API_KEY`), Meituan LongCat (`LONGCAT_API_KEY`), PackyCode (`PACKYCODE_API_KEY`), Shengsuanyun (`SHENGSUANYUN_API_KEY`), and Kimi For Coding (`KIMI_API_KEY`, `KIMICODE_API_KEY`). All except Kimi For Coding use the `openai_chat` provider type; Kimi For Coding speaks the `anthropic` protocol against `https://api.kimi.com/coding/`. Several presets default to China-region endpoints for domestic users: Moonshot uses `https://api.moonshot.cn/v1`, Z.AI/GLM uses `https://open.bigmodel.cn/api/paas/v4`, and the Ark, Qianfan, SiliconFlow, StepFun, LongCat, PackyCode, and Shengsuanyun presets point at their mainland hosts; a saved `baseUrl` override via `PATCH /provider/:id/settings` still wins over any preset default. Every built-in definition declares `env` as an ordered array of environment variable aliases; the connected check and worker credential resolution scan the aliases in order and use the first variable that is set. `PUT /provider/custom` validates and creates one (`name`, an `anthropic | openai | openai_chat` `providerType`, and a valid http(s) `baseUrl` are required; optional `contextWindow`/`maxTokens` must be positive integers; `id` defaults to a slug of the name and colliding with a built-in or existing custom id returns 409; an optional `apiKey` is stored under `state.providers[id]`); `DELETE /provider/custom/:id` removes a custom provider and its saved settings (built-ins return 400). `GET /provider` serves the SDK `provider.list()` contract `{ all, connected, default }` where `all` matches the `/config/providers` entries and `connected` lists providers with usable credentials (saved apiKey or environment variable). The `source`, `GET/PATCH settings`, and `PUT/DELETE /auth/:id` routes accept custom ids as well as built-ins; unknown ids still 404. `PATCH /provider/:id/settings` also accepts `contextWindow`/`maxTokens`: a positive integer stores an override in `state.providers[id]` that wins over the catalog (and, for custom providers, over the creation-time limits on the definition), while `null`, `0`, or `''` deletes the override and restores catalog resolution; `GET` settings echoes the effective values (`null` when unset).

Human-in-the-loop behavior is configured through two adapter-owned keys in `state.config` (written via `PATCH /config`, which merge-writes arbitrary keys without a whitelist): `_fe_hitlMode` (`'ask' | 'smart' | 'auto'`, default `'ask'`) and `_fe_hitlReviewModel` (string or `null`, default `null`, meaning the worker reuses the run's provider model for smart-mode gray-zone review). Every worker request — both `run` and `resume_interrupt` — carries the normalized snapshot as `hitlMode` (unknown or missing values normalize to `'ask'`) and `hitlReviewModel` (non-strings normalize to `null`), plus `hitlAllowlistPath` pointing at `<stateDir>/hitl-allowlist.json`, the persistent user allowlist described below. The semantics are enforced inside the PHP worker: `ask` keeps the current every-action interrupt behavior, `smart` lets the worker classify actions by rule and auto-decide low-risk ones with optional model review, and `auto` suppresses tool interrupts while `AskUserQuestion` still asks.

Sandbox execution is configured through five adapter-owned keys in `state.config` (same `PATCH /config` merge path): `_fe_sandboxEnabled` (boolean, default `false`), `_fe_sandboxBaseRootfs` (string path or `null`, default `null`), `_fe_sandboxNetwork` (`'blocked' | 'allow-all'`, default `'blocked'`), `_fe_sandboxMemoryMb` (integer, default `4096`), and `_fe_sandboxCpuCount` (integer, default `4`). Every worker request carries a `sandbox` object built by `buildSandboxRequest()`: `{ enabled: true, provider: 'tokimo', baseRootfs, network, memoryMb, cpuCount }` when enabled and provisioned, otherwise `{ enabled: false }`. The worker normalizes that into a `HaoCode\Sdk\Sandbox\SandboxConfig::tokimo(...)` and passes it to `HaoCodeConfig::sandbox`; with it set, the SDK reroutes `Read`/`Write`/`Glob`/`Grep`/`Bash` into the VM and the smart-HITL decider can auto-approve contained actions (emitting `auto_decision` `source: 'sandbox'`, reason `sandbox:contained: …`). `permissionMode` stays on `bypass_permissions`: the sandbox reroutes tool I/O, the permission check is orthogonal and still answered by `_fe_hitlMode`. The runner binary path is **not** part of the request — it flows through `HAOCODE_SANDBOX_BINARY` (Electron sets it at spawn time; dev leaves it unset so the SDK's `SandboxBinaryResolver` walks its dev-path / user-cache chain). Fail-closed behavior: when `_fe_sandboxEnabled` is `true` but `_fe_sandboxBaseRootfs` is missing or empty, `buildSandboxRequest` returns `{ enabled: false }` so the run continues unsandboxed instead of throwing; a present `baseRootfs` pointing at a nonexistent directory, or any SDK sandbox startup failure (missing Hyper-V, unsupported platform, 30s startup timeout), surfaces through the worker's existing error channel as `{ type: 'error', error, _fe_exception: 'RuntimeException' | 'InvalidArgumentException' }`.

Provisioning endpoints. The tokimo guest kernel + rootfs are release assets that the SDK ships outside the Composer archive, so first-use requires downloading them. `POST /sandbox/prepare` invokes the SDK's own installer (`vendor/bin/hao-code-sandbox install --with-runtime`, resolved from `HAOWORK_HAOCODE_AUTOLOAD`) with a 10-minute ceiling, single-flight per server to dedupe concurrent requests. On success it scans the SDK's guest-artifact cache (`<cacheRoot>/<kernel-tag>+<rootfs-tag>/<platform>-<arch>/base`, where `cacheRoot` honors `HAOCODE_SANDBOX_CACHE`/`LOCALAPPDATA`/`XDG_CACHE_HOME`) and writes the resolved `base` directory to `state.config._fe_sandboxBaseRootfs` so subsequent worker requests enable the sandbox. The call returns `{ ok: true, baseRootfs }` or, on failure, `503 { ok: false, error }` (installer-unavailable, download checksum mismatch, platform unsupported). `GET /sandbox/status` exposes the current state without side effects: `{ supported, installerAvailable, preparing, installedRootfs, configuredRootfs, configuredMissing }` — `configuredMissing` is `true` when `_fe_sandboxBaseRootfs` no longer exists on disk (user cleared the cache), which the UI surfaces as "runtime needs reinstall".

In `smart` mode the worker emits one `auto_decision` JSON-lines event per auto-decided action before continuing in the same process (`{ type: 'auto_decision', sessionId, interruptId, actionId, toolName, toolInput, decision: 'approve' | 'reject', source: 'rule' | 'review' | 'sandbox', riskLevel: 'low' | 'medium' | 'high' | 'critical', reason }`). A `source: 'rule'` decision comes from the SDK's rule grading (including user-allowlist hits, whose reason reads `allowlist:user_rule: …`), a `source: 'review'` decision from the optional gray-zone model review, and a `source: 'sandbox'` decision from an action the SDK proved would execute inside a configured sandbox (reason reads `sandbox:contained: …`). The compatibility server translates each one into an adapter-owned SSE event `permission.auto_resolved` with properties `{ sessionID, requestID: 'req_' + actionId, permission: toolName, metadata: { input, description: reason, _fe_interruptId, _fe_actionId, _fe_autoDecision, _fe_source, _fe_riskLevel } }`, and appends an audit record to `state.autoDecisions[sessionId]` (FIFO-capped at 100 entries per session, persisted in runtime-state.json). These events never create pending permission or question cards; unknown event fields are normalized conservatively (unknown decisions display as `reject`, unknown risk levels as `high`). The read-only `GET /auto-decisions?sessionID=…` route returns one session's persisted history (directory stripped) so the UI can backfill its resolved-decision cards after a refresh or reconnect; the records are also removed when their session is deleted. When a smart-mode batch escalates to a human, the worker additionally emits one `auto_decision` event per action with `decision: 'escalate'`, `source: 'rule' | 'review' | 'batch' | 'sandbox'`, and a machine-readable `reason` (e.g. `rule:red_line:…`, `rule:ask:…`, `review:unavailable:…`, `review:unsure`, `batch:escalated`, `batch:circuit_breaker`) before forwarding the interrupt unchanged. The compatibility server does not persist escalations to `state.autoDecisions` and does not emit `permission.auto_resolved` for them; instead it stashes them by `interruptId:actionId` and merges `_fe_escalationReason`, `_fe_escalationSource`, and `_fe_escalationRisk` into the resulting pending permission's metadata so `PermissionCard` can explain why human confirmation is required. Replying **Always Allow** (`reply: 'always'`) to a pending Bash permission also persists codex-style rules for later runs: the server derives one or more rules from the trimmed command and appends them to `<stateDir>/hitl-allowlist.json` (shape `{ "version": 2, "rules": [...] }` with entries `{ "type": "prefix", "tokens": [...], "addedAt": <ISO 8601>, "source": "user" }` or `{ "type": "exact", "command": "...", "addedAt": <ISO 8601>, "source": "user" }`; deduped by rule type and content, writes serialized with the other state writes, missing or corrupted files rebuilt as an empty `version: 2` document, non-Bash tools never written; legacy `version: 1` `{ "command", ... }` entries are read back unchanged, count as exact rules for dedup, and remain in the file untouched). Rule derivation: a command containing a heredoc (`<<`) is stored as one whole-command exact rule; otherwise the command is split on `&&`/`||`/`;`/`|` (never inside quotes, never on newlines), leading `VAR=value` assignments are stripped from each segment, and each non-empty segment yields one rule — an exact rule of the full segment when it contains a file redirection (`>`, `>>`, `2>`, or a `tee` token, excluding `2>&1` and `>/dev/null`), when its first token contains a `/` (path execution), or when its first token is a powerful interpreter or network tool (sudo/su/doas, python/node/bash/sh/zsh/fish/php/ruby/perl/lua, env/xargs/eval/exec/source/`.`, curl/wget/ssh/scp/rsync/nc/ncat/osascript/pwsh/powershell); a prefix rule of the first two tokens for subcommand tools (git/npm/yarn/pnpm/bun/composer/cargo/pip/pip3/docker/kubectl/helm/go/gem/brew), extended to three tokens when the second token is `run`/`exec`/`dlx`; otherwise a prefix rule of the first token. Every worker request carries that path as `hitlAllowlistPath`, and the SDK matches trimmed Bash commands against it before rule grading (whole-string exact match first, otherwise per-segment matching where every segment must hit at least one rule) — an allowlist hit approves the action with `source: 'rule'` and reason `allowlist:user_rule: …`, so a persisted rule can exempt a command from later interrupts and even red-line grading. The terminal-event contract is unchanged: the worker still finishes every run with exactly one of `result`, `error`, or `interrupt`.

Shutdown is a drain operation: the compatibility server terminates workers, waits for active session pipelines, and flushes the serialized state queue before callers may remove its data directory. Store initialization and writes are serialized so concurrent sessions cannot race the initial state load or share a temporary state file.

Do not leak HaoCode PHP objects into the browser contract. Adapter-only response fields must use the `_fe_` prefix. Secrets stay in the local runtime-state file with mode `0600` and are never returned by API responses.
