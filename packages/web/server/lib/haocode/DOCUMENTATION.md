# HaoCode compatibility runtime

This module keeps the OpenChamber browser contract stable while replacing the upstream OpenCode process with HaoCode.

- `store.js` persists adapter-owned sessions, OpenCode-shaped message snapshots, pending interrupts, provider settings, and status.
- `worker-supervisor.js` owns one short-lived PHP process per active session. The PHP process is not an HTTP sidecar; it reads one request over stdin and emits JSON-lines events over stdout.
- `compat-server.js` exposes the subset of the OpenCode HTTP/SSE protocol required by the existing UI and translates HaoCode worker events into OpenCode session events.

Project and user Agent/Command definitions remain stored in OpenCode-compatible config files so the inherited settings UI can edit them. The adapter lists those definitions through `/agent` and `/command`; a selected custom agent's prompt is appended to the corresponding HaoCode run. MCP definitions are converted to a private, generated HaoCode settings file for each project and enabled for that worker run. `/mcp` reports whether each definition is enabled and structurally usable; the actual connection is established by HaoCode when a run starts.

HaoCode `TodoWrite` events are persisted per session and published as `todo.updated`. Session diff, VCS diff/status, and file status responses are derived from the selected workspace's current Git worktree. They describe current workspace changes, including changes that predate the session; they are not an attribution ledger. HaoCode's LSP client is created only while the `LSP` tool runs, so `/lsp` correctly has no persistent server process to report even though the tool is available to the agent.

Message history pagination preserves complete renderable turns: a limited page includes any older parent user messages referenced by assistant records in that page, and `x-next-cursor` identifies the next older base page. Supplemental parent records may therefore make a response slightly larger than its requested limit.

Model limits have one runtime source of truth. The compatibility server resolves each model's context and output limits from the shared models.dev catalog, exposes those values through `/config/providers`, and sends the same values to the PHP worker. When the catalog is unavailable and no stale cache exists, it falls back to HaoCode's `200000` context and `16384` output defaults. The worker applies the resolved context through `HAOCODE_CONTEXT_WINDOW`, so the provider page, Context sidebar, and HaoCode compaction budget use the same limits.

Built-in Anthropic and OpenAI support OAuth subscription login alongside API keys; `GET /provider/auth` advertises one `{ type: 'oauth' }` method for each (DeepSeek and custom providers stay api-key only). Both flows use PKCE (S256) implemented with `node:crypto` — no extra dependencies. Anthropic uses the manual paste-code flow (`method: 'code'`): `POST /provider/anthropic/oauth/authorize` returns the `claude.ai/oauth/authorize` URL whose `state` doubles as the PKCE verifier, and `POST /provider/anthropic/oauth/callback` accepts the pasted `"<code>#<state>"` (or a bare code, which reuses the pending state), exchanges it at `console.anthropic.com/v1/oauth/token`, and stores the result. OpenAI uses the browser flow (`method: 'auto'`): authorize starts a `node:http` localhost redirect listener (ports 1455-1465 tried in order), and the callback route awaits the browser redirect (120s timeout), validates the returned state, exchanges the code at `auth.openai.com/oauth/token`, and extracts `accountId` from the `chatgpt_account_id` JWT claim when present. Tokens persist as `state.providers[id].oauth = { access, refresh, expires (ms epoch), accountId? }` in the mode-`0600` runtime-state file and are never returned by API responses; pending flows live only in memory and expire after 10 minutes. Credential resolution for worker runs is oauth access token → saved apiKey → environment variable: a token within 5 minutes of expiry is refreshed synchronously through the provider's `refresh_token` grant first (single-flight per provider, written back to state; a failed refresh falls back to the saved apiKey or environment). OAuth client ids default to the public OpenCode clients and can be overridden through `HAOWORK_OAUTH_ANTHROPIC_CLIENT_ID` / `HAOWORK_OAUTH_OPENAI_CLIENT_ID`. When the resolved credential is an Anthropic OAuth access token the worker request marks it `oauthBearer: true`, which `worker.php` forwards as `HAOCODE_OAUTH_BEARER=1` because `HaoCodeConfig` has no such constructor field. `DELETE /auth/:id` removes both the saved apiKey and the oauth record and cancels any pending flow for that provider.

The provider registry merges built-in definitions (Anthropic, OpenAI, DeepSeek) with user-defined custom providers persisted in `state.customProviders[id] = { name, providerType, baseUrl, env, models, contextWindow?, maxTokens?, createdAt }`. `PUT /provider/custom` validates and creates one (`name`, an `anthropic | openai | openai_chat` `providerType`, and a valid http(s) `baseUrl` are required; optional `contextWindow`/`maxTokens` must be positive integers; `id` defaults to a slug of the name and colliding with a built-in or existing custom id returns 409; an optional `apiKey` is stored under `state.providers[id]`); `DELETE /provider/custom/:id` removes a custom provider and its saved settings (built-ins return 400). `GET /provider` serves the SDK `provider.list()` contract `{ all, connected, default }` where `all` matches the `/config/providers` entries and `connected` lists providers with usable credentials (saved apiKey or environment variable). The `source`, `GET/PATCH settings`, and `PUT/DELETE /auth/:id` routes accept custom ids as well as built-ins; unknown ids still 404. `PATCH /provider/:id/settings` also accepts `contextWindow`/`maxTokens`: a positive integer stores an override in `state.providers[id]` that wins over the catalog (and, for custom providers, over the creation-time limits on the definition), while `null`, `0`, or `''` deletes the override and restores catalog resolution; `GET` settings echoes the effective values (`null` when unset).

Human-in-the-loop behavior is configured through two adapter-owned keys in `state.config` (written via `PATCH /config`, which merge-writes arbitrary keys without a whitelist): `_fe_hitlMode` (`'ask' | 'smart' | 'auto'`, default `'ask'`) and `_fe_hitlReviewModel` (string or `null`, default `null`, meaning the worker reuses the run's provider model for smart-mode gray-zone review). Every worker request — both `run` and `resume_interrupt` — carries the normalized snapshot as `hitlMode` (unknown or missing values normalize to `'ask'`) and `hitlReviewModel` (non-strings normalize to `null`), plus `hitlAllowlistPath` pointing at `<stateDir>/hitl-allowlist.json`, the persistent user allowlist described below. The semantics are enforced inside the PHP worker: `ask` keeps the current every-action interrupt behavior, `smart` lets the worker classify actions by rule and auto-decide low-risk ones with optional model review, and `auto` suppresses tool interrupts while `AskUserQuestion` still asks.

In `smart` mode the worker emits one `auto_decision` JSON-lines event per auto-decided action before continuing in the same process (`{ type: 'auto_decision', sessionId, interruptId, actionId, toolName, toolInput, decision: 'approve' | 'reject', source: 'rule' | 'review' | 'sandbox', riskLevel: 'low' | 'medium' | 'high' | 'critical', reason }`). A `source: 'rule'` decision comes from the SDK's rule grading (including user-allowlist hits, whose reason reads `allowlist:user_rule: …`), a `source: 'review'` decision from the optional gray-zone model review, and a `source: 'sandbox'` decision from an action the SDK proved would execute inside a configured sandbox (reason reads `sandbox:contained: …`). The compatibility server translates each one into an adapter-owned SSE event `permission.auto_resolved` with properties `{ sessionID, requestID: 'req_' + actionId, permission: toolName, metadata: { input, description: reason, _fe_interruptId, _fe_actionId, _fe_autoDecision, _fe_source, _fe_riskLevel } }`, and appends an audit record to `state.autoDecisions[sessionId]` (FIFO-capped at 100 entries per session, persisted in runtime-state.json). These events never create pending permission or question cards; unknown event fields are normalized conservatively (unknown decisions display as `reject`, unknown risk levels as `high`). The read-only `GET /auto-decisions?sessionID=…` route returns one session's persisted history (directory stripped) so the UI can backfill its resolved-decision cards after a refresh or reconnect; the records are also removed when their session is deleted. When a smart-mode batch escalates to a human, the worker additionally emits one `auto_decision` event per action with `decision: 'escalate'`, `source: 'rule' | 'review' | 'batch' | 'sandbox'`, and a machine-readable `reason` (e.g. `rule:red_line:…`, `rule:ask:…`, `review:unavailable:…`, `review:unsure`, `batch:escalated`, `batch:circuit_breaker`) before forwarding the interrupt unchanged. The compatibility server does not persist escalations to `state.autoDecisions` and does not emit `permission.auto_resolved` for them; instead it stashes them by `interruptId:actionId` and merges `_fe_escalationReason`, `_fe_escalationSource`, and `_fe_escalationRisk` into the resulting pending permission's metadata so `PermissionCard` can explain why human confirmation is required. Replying **Always Allow** (`reply: 'always'`) to a pending Bash permission also persists codex-style rules for later runs: the server derives one or more rules from the trimmed command and appends them to `<stateDir>/hitl-allowlist.json` (shape `{ "version": 2, "rules": [...] }` with entries `{ "type": "prefix", "tokens": [...], "addedAt": <ISO 8601>, "source": "user" }` or `{ "type": "exact", "command": "...", "addedAt": <ISO 8601>, "source": "user" }`; deduped by rule type and content, writes serialized with the other state writes, missing or corrupted files rebuilt as an empty `version: 2` document, non-Bash tools never written; legacy `version: 1` `{ "command", ... }` entries are read back unchanged, count as exact rules for dedup, and remain in the file untouched). Rule derivation: a command containing a heredoc (`<<`) is stored as one whole-command exact rule; otherwise the command is split on `&&`/`||`/`;`/`|` (never inside quotes, never on newlines), leading `VAR=value` assignments are stripped from each segment, and each non-empty segment yields one rule — an exact rule of the full segment when it contains a file redirection (`>`, `>>`, `2>`, or a `tee` token, excluding `2>&1` and `>/dev/null`), when its first token contains a `/` (path execution), or when its first token is a powerful interpreter or network tool (sudo/su/doas, python/node/bash/sh/zsh/fish/php/ruby/perl/lua, env/xargs/eval/exec/source/`.`, curl/wget/ssh/scp/rsync/nc/ncat/osascript/pwsh/powershell); a prefix rule of the first two tokens for subcommand tools (git/npm/yarn/pnpm/bun/composer/cargo/pip/pip3/docker/kubectl/helm/go/gem/brew), extended to three tokens when the second token is `run`/`exec`/`dlx`; otherwise a prefix rule of the first token. Every worker request carries that path as `hitlAllowlistPath`, and the SDK matches trimmed Bash commands against it before rule grading (whole-string exact match first, otherwise per-segment matching where every segment must hit at least one rule) — an allowlist hit approves the action with `source: 'rule'` and reason `allowlist:user_rule: …`, so a persisted rule can exempt a command from later interrupts and even red-line grading. The terminal-event contract is unchanged: the worker still finishes every run with exactly one of `result`, `error`, or `interrupt`.

Shutdown is a drain operation: the compatibility server terminates workers, waits for active session pipelines, and flushes the serialized state queue before callers may remove its data directory. Store initialization and writes are serialized so concurrent sessions cannot race the initial state load or share a temporary state file.

Do not leak HaoCode PHP objects into the browser contract. Adapter-only response fields must use the `_fe_` prefix. Secrets stay in the local runtime-state file with mode `0600` and are never returned by API responses.
